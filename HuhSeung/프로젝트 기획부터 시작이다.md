1. 기획이 왜 필요한가?
- 방향성
    - 팀원 마다 생각이 다르면 시간 낭비 = 프로젝트 실패
    - 프로젝트가 나아갈 방향을 일치
- 경계선
    - 프로젝트에서 무엇을 하지 않을지 결정하는 것
    - 기능을 너무 많이 잡으면 완성 불가
    - 목표가 모호하면 구현 우선순위가 뒤죽박죽
    - 끝나는 기준이 없으면 끝나지 않는 프로젝트

무엇을 만들 것인지, 어디까지 만들 것인지

1. 기획에 포함되면 좋은 내용
- 문제 정의 → 대상자 정의 → 사용 시나리오 → 기능 정의 → 화면 단위 구체화
    1. 문제 정의
        1. 이 프로젝트는 누구의 어떤 문제를 해결하는가?
        2. 이게 없으면 무엇이 불편한가?
        3. ex) 파킨슨 환자(대상)들이 높은 비용(문제점)으로 재활 치료를 받는 데 어려움이 있다.
    2. 사용자 정의
        1. 사용자 → 구체적인 1인
        2. 상황 → 언제 쓰는가
        3. 불편 → 지금 뭐가 문제인가
        4. 목표 → 무엇을 얻고 싶은가
        5. 사용자 페르소나
    3. 사용 시나리오
        1. 사용자가 이 서비스를 어떻게 처음부터 끝까지 사용하는가?
        2. 시연의 대한 고민도 필요!!
        3. 실제 발표시에도 스토리를 녹여내야 한다.
    4. 기능 정의
        1. 필요한 기능을 나열하기
        2. 나열한 기능들을 3단계로 분리하기(우선 순위 Must, Should, Could)
        3. MVP는 반드시 구현, 다른 기능은 과감히 포기
    5. 화면 단위 / 기능 단위 쪼개기
    
    1. 기술 스택 & 역할 분담
        1. 기술 스택 선정
        2. 역할 분담
        3. 역할은 메인과 서브로

기획 문서에서 팀원들의 생각 공유가 중요, 필요하면 언제든 수정 가능(대신 이유가 있어야 함)

### 인사이트 정리

- 기획은 팀의 **공동 방향성**을 맞추고, 프로젝트의 **경계(하지 않을 것)**를 명확히 하는 작업이다.
- 목표와 범위가 모호하면 기능 우선순위가 뒤섞이고, 끝나는 기준이 없어 **끝나지 않는 프로젝트**가 되기 쉽다.
- "무엇을 만들 것인지, 어디까지 만들 것인지"라는 질문이 기획에서 가장 중요한 핵심 질문이다.
- 문제 정의 → 사용자 정의 → 사용 시나리오 → 기능 정의 → 화면 단위 구체화 → 기술 스택 & 역할 분담 흐름은 **아이디어부터 개발 직전까지를 한 흐름으로 묶은 기획 프로세스**이다.
- 사용자 정의에서 한 명의 구체적인 사용자, 사용 상황, 불편, 목표를 설정하는 것은 **사용자 중심·스토리 중심 기획**을 가능하게 한다.
- 기능 정의에서 Must / Should / Could를 나누고, MVP만 반드시 구현하겠다는 원칙은 **리소스를 고려한 현실적인 우선순위 설정**에 대한 기준이 된다.
- 기획 문서는 고정된 답이 아니라, 이유를 가지고 수정해 가는 **팀 공동의 살아있는 기준점**이라는 태도가 중요하다.

# 소프트웨어 아키텍처 설계 가이드

프로젝트 시작 전, 탄탄한 설계를 위해 반드시 짚고 넘어가야 할 아키텍처 설계의 핵심 요소들을 정리했습니다.

---

## 1. 소프트웨어 아키텍처란?
소프트웨어를 구성하는 요소들을 정의하고, 이들이 어떻게 상호작용할지를 설계하는 근본적인 구조입니다.

### 왜 아키텍처가 중요한가?
*   **초기 설계의 결정적 역할:** 프로젝트 초기에 내린 결정들이 서비스의 안정성과 확장성을 좌우합니다.
*   **커뮤니케이션의 기준점:** 팀원 간 기술적 의사소통의 공통된 기반이 됩니다.

---

## 2. MSA (Microservices Architecture) 도입 고려
단순히 유행을 따르는 것이 아니라, 제대로 된 이해를 바탕으로 우리 프로젝트에 적용해야 합니다. MSA를 성공적으로 도입하기 위해서는 다음과 같은 단계적 접근이 필요합니다.

### 요구사항 분석 프로세스
1.  **이해관계자 선별:** 시스템에 영향을 주거나 받는 주요 주체들을 확인합니다.
2.  **기능 요구사항 정의:** 시스템이 '무엇'을 해야 하는지 명확히 규정합니다.
3.  **비기능 요구사항 정의:** 성능, 보안, 확장성 등 시스템의 '품질'적 측면을 정의합니다.

---

## 3. 아키텍처 도식화 (Diagramming) 가이드
우리 프로젝트의 구조를 시각화할 때, 흔히 저지르는 실수와 지향해야 할 방향을 정리했습니다.

| 구분 | 특징 | 피드백 |
| :--- | :--- | :--- |
| **사례 1** | 단순히 기술 스택만 나열한 경우 | 기술 스택의 단순 열거는 구조 파악에 도움이 되지 않으므로 지양해야 합니다. |
| **사례 2** | 관계(Relation) 설명이 부족한 경우 | 각 구성 요소 간의 연결 고리가 없어 시스템 흐름을 파악하기 어렵습니다. |
| **사례 3** | 기술 스택 중심의 관계도 | 릴레이션은 있으나 여전히 기술 스택에만 치중되어 있어 아키텍처적 깊이가 아쉽습니다. |
| **참고 사례** | 아키텍처 구성 요소와 관계의 조화 | **이 방식이 가장 권장됩니다.** 구성 요소와 상호작용이 명확히 드러나야 합니다. |
